# plan_v8.md — Falsification Engine v8 Implementation Plan

## Enhancement 6: Theory Package Structure Migration

### Design Principle

**This is a plumbing change, not a theory change.**

The economic logic is identical. The pipeline architecture is identical. The scoring arithmetic is identical. The engine reads the same information from a better-organised source. Every component that reads theory content must be updated to read from the new four-file package structure instead of the monolithic `THEORY_MODULE_{theory_id}_v1.md` files.

v8 preserves behavior, simplifies interfaces, and avoids speculative prompt-budget engineering.

### Prerequisite

**The theory registry reorganisation must be complete.** All 8 theory directories must exist with CORE.md / ACTIVATION.md / TACTICAL.md / PLAYBOOK.md, plus INTERACTION_MATRIX.md at the registry level. The reorganisation was executed per `THEORY_REORGANISATION_SPEC.md`. The old monolithic files remain in `theories/old_format/` as archived originals.

---

## What Exists (v1 through v7 — all complete or in build)

The full pipeline through v7:

```
Pass 1:   Activation scoring — 8 modules scored independently → Active / Adjacent / Inactive
Pass 1.5: Regime annotation — compute_regime_flags() from activation results → active flags list
Pass 2:   LLM generation — active theories + regime context + channel tags
            + prior hypothesis thread context (v7)
            + structured payoff band output requirement (v6)
            + continuation lineage + lifecycle taxonomy (v7)
            → 7-9 hypotheses with CONFIRM/UPDATE/RENEW/RETIRE/NEW actions
Pass 3:   LLM elimination — adversarial attack + sector falsifier appendices
            + channel verification + falsifier lifecycle (v7)
            + emergent risk slot (v7)
            → SURVIVED / WOUNDED / KILLED + corrected channels + sector audit trail
Pass 4:   Mechanical conviction scoring (zero-LLM):
            Stage 1: RAW = S(0.30) + Q(0.30) + C(0.25) + F(0.15)
            Stage 2: DISCOUNTED = RAW × D_f × D_u × D_r + overlap_adjustment
            Stage 3: FINAL = min(SCORE, horizon_cap, expression_cap, realization_cap) → max(FINAL, 5.0)
Pass 4.5: Realization overlay (zero-LLM, post-conviction)
Pass 5:   Human decision layer — system never recommends trades or sizes
```

**Current theory file structure (what v8 replaces):**

```
theories/
  THEORY_MODULE_structural_fragility_v1.md      ← monolithic, ~2000 lines
  THEORY_MODULE_fiscal_dominance_liquidity_v1.md
  ... (8 files total)
```

Each monolithic module mixes invariant theory, activation logic, market expressions, and system instructions into one document. Every pass reads the same file and extracts what it needs.

## What v8 Changes

### New Theory File Structure

```
theories/
  old_format/                          ← archived originals, nothing reads from here
    THEORY_MODULE_structural_fragility_v1.md
    ... (8 files)
  structural_fragility/
    CORE.md          — invariant theory, causal mechanism, deep falsifiers, scope limits
    ACTIVATION.md    — phases, scored indicators, thresholds, weights, data ownership, severity assignments, context flags
    TACTICAL.md      — asset expressions, ETF mappings, sector depth, directional predictions, expression monitors
    PLAYBOOK.md      — generator guidance, evaluator checks, composition rules, failure modes
  fiscal_dominance_liquidity/
    CORE.md
    ACTIVATION.md
    TACTICAL.md
    PLAYBOOK.md
  ... (6 more theory directories, same structure) ...
  INTERACTION_MATRIX.md    ← single source of truth for cross-theory relationships
  REGISTRY_INDEX.md        ← mechanical summary, generated by loader (see below)
```

### What Each Pass Reads — Before and After

| Pass | Before (monolithic) | After (packaged) |
|------|-------------------|-----------------|
| Pass 1 | Full module → parse activation section | ACTIVATION.md only |
| Pass 1.5 | Unchanged — reads activation results, not theory files | Unchanged |
| Pass 2 | Full module | CORE.md + TACTICAL.md + PLAYBOOK.md + INTERACTION_MATRIX.md (new) |
| Pass 3 | Full module | CORE.md + ACTIVATION.md + TACTICAL.md + PLAYBOOK.md + INTERACTION_MATRIX.md (new) |
| Pass 4 | Full module → parse severity assignments | ACTIVATION.md only (severity assignments section) |
| Pass 4.5 | Unchanged — reads realization data, not theory files | Unchanged |

**Key difference from monolithic:** Each pass gets the **full files** it needs. No within-file section filtering. The current system sends the full monolithic module to each pass; the migration sends the equivalent content assembled from the relevant package files. Prompt budget optimisation (section-level filtering, content reduction for Adjacent theories) is explicitly deferred.

## Component 1: Theory Package Loader

### The Problem

The system currently loads one file per theory: `theories/THEORY_MODULE_{theory_id}_v1.md`. Every component that reads theory content — activation scoring, prompt building, conviction scoring — reads from this single file and extracts what it needs via section parsing.

### The Solution

A new theory package loader that understands the directory structure. It replaces the monolithic file reader with a directory-based reader that loads four files per theory.

### Loader Contract

The loader reads the theory directory and produces a structured theory package object per theory. Every downstream consumer reads from this object, never from raw markdown files.

```
TheoryPackage:
  theory_id: string
  core: string          — full text of CORE.md
  activation: string    — full text of ACTIVATION.md
  tactical: string      — full text of TACTICAL.md
  playbook: string      — full text of PLAYBOOK.md
  falsifier_registry: FalsifierEntry[]  — pre-joined (see Component 2)
  data_ownership: IndicatorOwnership[]  — parsed from ACTIVATION.md (see Component 5)
  context_flags: ContextFlag[]          — parsed from ACTIVATION.md (see Component 5)
```

### Loading Behavior

- Reads all four files at startup. Not lazy-loaded per pass. The theory registry is 8 theories × 4 files = 32 files. This is small.
- Missing file handling: if any of the four files is missing for a theory, the loader raises a hard error. All four files are required per the reorganisation spec. Partial packages are not valid.
- The loader does NOT parse markdown into structured fields beyond what is specified in Components 2 and 5. The prompt builder receives the raw markdown text of each file and includes it in prompts as-is.

### Adapter Layer

During migration, the loader also provides an adapter that maps the new TheoryPackage to the old monolithic format. This allows old-format and new-format to run in parallel for validation (see Component 6).

The adapter concatenates the four files in a canonical order (CORE → ACTIVATION → TACTICAL → PLAYBOOK) with section separators. The result is functionally equivalent to the old monolithic module. This is temporary scaffolding, removed after cutover.

## Component 2: Falsifier Registry Builder

### The Problem

In the old monolithic format, falsifier conditions and their severity assignments lived in the same file. In the new structure, conditions are defined in CORE.md (`deep_falsifiers`) and severity is assigned in ACTIVATION.md (`falsifier_severity_assignments`). The evaluator in Pass 3 and the conviction pipeline in Pass 4 both need the joined information.

### The Solution

The loader pre-joins falsifiers at load time into a canonical falsifier registry object per theory, keyed by `falsifier_id`.

```
FalsifierEntry:
  falsifier_id: string       — e.g., "H1", "H2", "S1", "S3"
  condition: string          — from CORE.md deep_falsifiers
  logic: string              — from CORE.md deep_falsifiers
  classification: string     — "hard" or "soft" — from ACTIVATION.md
  severity: string | null    — "minor" / "medium" / "major" for soft; null for hard — from ACTIVATION.md
  discount: number | null    — 0.10 / 0.25 / 0.45 for soft; null for hard — from ACTIVATION.md
```

### Validation Rules (enforced at load time)

1. Every falsifier ID in CORE.md `deep_falsifiers` must have exactly one entry in ACTIVATION.md `falsifier_severity_assignments`. No orphans in CORE.md.
2. Every falsifier ID in ACTIVATION.md `falsifier_severity_assignments` that references a CORE.md falsifier must exist in CORE.md. No orphans in ACTIVATION.md.
3. State-level falsifiers defined in ACTIVATION.md (those not referencing CORE.md) are self-contained — condition and severity both in ACTIVATION.md. These are also included in the registry.
4. Validation failure on any theory is a hard loader error. The system does not start with an inconsistent falsifier registry.

### Downstream Consumption

- **Pass 3 (evaluator):** Receives the joined falsifier registry as a structured section in the elimination prompt. The evaluator uses the `condition` and `logic` fields for attack reasoning. The `severity` field rides along for audit completeness but the evaluator does not use it in reasoning — the evaluator's job is status assignment (CLEAR / TRIGGERED / UNTESTABLE), not discount application.
- **Pass 4 (conviction):** Reads `severity` and `discount` fields from the registry to apply mechanical discounts. No change to the discount arithmetic (minor=0.10, medium=0.25, major=0.45).

## Component 3: INTERACTION_MATRIX Loader

### The Problem

No current engine component reads a cross-theory interaction file. The INTERACTION_MATRIX.md is a new input introduced by the reorganisation. The engine must consume it in Pass 2 (generation) and Pass 3 (elimination).

### The Solution

A matrix loader that parses INTERACTION_MATRIX.md and provides filtered views based on current activation status.

### Loading Behavior

- Parsed at startup into a structured list of pairwise interaction entries.
- Each entry contains: theory_a, theory_b, relationship, invariant_logic, expression_detail_location.
- Shared Upstream Cause Warnings parsed as a separate list: shared_cause, theories_affected, discounting_note.

### Filtering

Before injection into prompts, the matrix is filtered to **rows where at least one theory is Active**. This includes Active × Active and Active × Adjacent rows. Rows where both theories are Inactive or Adjacent are excluded.

Shared Upstream Cause Warnings are filtered to entries where at least two of the affected theories are Active.

### Pass-Specific Role

- **Pass 2 (generation):** The filtered matrix enters the generation prompt as composition guidance. It tells the generator which theory combinations are valid, what specificity gain each combination produces, and which compositions are prohibited or low-value.
- **Pass 3 (elimination):** The filtered matrix enters the elimination prompt as composition validation criteria. It tells the evaluator how to check whether the generator combined theories correctly — did the combination narrow the prediction, or is it narrative padding? It also provides the Shared Upstream Cause Warnings so the evaluator can flag double-counting.
- **Pass 4:** The matrix does NOT enter conviction scoring in v8. If interaction-based conviction adjustment is ever implemented, that is a later enhancement.

## Component 4: Pass-Specific Prompt Composition

### The Problem

The prompt builder currently injects the full monolithic module for each theory into Pass 2 and Pass 3 prompts. It must be updated to compose prompts from the new file structure.

### The Solution

The prompt builder receives TheoryPackage objects (from the loader) and the filtered INTERACTION_MATRIX (from the matrix loader). It assembles pass-specific prompts by including the relevant files.

### Prompt Assembly Rules

**Pass 1 — Activation Scoring:**
- Active and Adjacent: reads `activation` field (full ACTIVATION.md text) from TheoryPackage.
- Inactive: skipped.
- No change to scoring arithmetic. Only the file source changes.

**Pass 2 — Generation:**
- For each Active theory: include `core` + `tactical` + `playbook` from TheoryPackage.
- For each Adjacent theory (at most one, per existing contract): include `core` + `tactical` + `playbook` from TheoryPackage.
- Include filtered INTERACTION_MATRIX (composition guidance).
- Inactive theories: nothing.

**Pass 3 — Elimination:**
- For each theory invoked by any surviving hypothesis: include `core` + `activation` + `tactical` + `playbook` from TheoryPackage, plus the pre-joined `falsifier_registry`.
- Include filtered INTERACTION_MATRIX (composition validation + Shared Upstream Cause Warnings).
- The `falsifier_registry` is included as a structured block, separate from the raw file text. The evaluator references the registry for attack logic, not the raw CORE.md and ACTIVATION.md separately.

**Pass 4 — Conviction Scoring:**
- Reads `falsifier_registry` from TheoryPackage for severity discounts. Specifically: the `discount` field on each FalsifierEntry where the evaluator assigned TRIGGERED status.
- No change to the discount arithmetic.

### Adjacent Theory Treatment

Adjacent theories receive the same files as Active theories for their relevant pass. The current system sends the full monolithic module for Adjacent theories. This migration preserves that behavior. Reducing Adjacent content for token budget reasons is deferred.

## Component 5: Data Ownership Enforcement

### The Problem

The reorganised ACTIVATION.md files classify every indicator by data ownership (`mechanical`, `computed-mechanical`, `web-search`, `qualitative`). The old monolithic modules did not make this classification explicit or enforceable. The reorganisation spec is clear: `qualitative` indicators must not be mechanically scored.

### The Solution

Three enforcement rules in v8, applied at the loader and activation scoring layers:

**Rule 1: Qualitative indicators must not be mechanically scored.**
The loader validates that no indicator in the `activation_table` section of ACTIVATION.md has data ownership `qualitative`. The reorganisation already moved qualitative indicators to `context_flags`, but the loader enforces this as a contract: if a `qualitative` indicator appears in an activation table, it is a loader error.

**Rule 2: Context flags are routed to the generator but excluded from Pass 1 scoring.**
The loader parses `context_flags` from ACTIVATION.md into a separate list on the TheoryPackage. Context flags enter the generation prompt (Pass 2) as supplementary qualitative information. They do NOT enter Pass 1 activation scoring. They do NOT have weights or thresholds. This separation already exists in the reorganised ACTIVATION.md files (context_flags is a separate section from activation_table), but v8 makes it a loader-enforced contract.

**Rule 3: Data ownership is surfaced as metadata, not used for fetch orchestration.**
The `data_ownership` field on each indicator is parsed and stored on the TheoryPackage. It is available for downstream consumers (including a future data agent that routes fetch strategy by ownership type). In v8, it is metadata — the data agent's fetch behavior is not rebuilt. Full fetch-strategy orchestration by data ownership is deferred.

```
IndicatorOwnership:
  indicator_name: string
  metric_source: string
  data_ownership: "mechanical" | "computed-mechanical" | "web-search" | "qualitative"
  dependencies: string[] | null   — for computed-mechanical only
```

## Component 6: Migration Validation

### The Problem

The migration must not change engine behavior. Activation scores, generation quality, and elimination quality must be preserved. But these operate at different levels of testability — activation is mechanical and regression-testable; generation and elimination are quality-sensitive and require human judgment.

### The Solution

Four validation layers, each with different automation levels.

### Layer 1: Activation Equivalence — Hard Automated Gate

Run the activation scoring pipeline on the same briefing packet, once with the old monolithic loader (via adapter) and once with the new package loader. Activation scores must be **identical** — same numeric score, same Active/Adjacent/Inactive status, same phase assignment for two-phase theories.

This is the hard gate. If activation scores diverge, the migration has a bug. No cutover until this passes.

### Layer 2: Loader Contract Tests — Automated

Unit tests validating the loader's structural integrity:

- Missing file detection: remove one file from a theory package, verify loader raises hard error.
- Falsifier orphan detection: add a falsifier ID to CORE.md without a severity in ACTIVATION.md, verify loader raises hard error. Reverse: add a severity in ACTIVATION.md referencing a non-existent CORE.md falsifier, verify loader raises hard error.
- Data ownership validation: place a `qualitative` indicator in an activation_table, verify loader raises hard error.
- Context flag separation: verify that context flags are not included in the scored indicator list.
- Phase handling: verify two-phase theories (structural_fragility, debt_cycle_short, capital_flows) produce per-phase activation tables correctly.
- INTERACTION_MATRIX filtering: verify that only rows with ≥1 Active theory survive filtering.

### Layer 3: Prompt Assembly Snapshot Tests — Automated

Snapshot tests that verify prompt composition correctness:

- Pass 1 prompt contains ACTIVATION.md content and nothing else per theory.
- Pass 2 prompt contains CORE.md + TACTICAL.md + PLAYBOOK.md per Active theory, plus filtered INTERACTION_MATRIX.
- Pass 3 prompt contains CORE.md + ACTIVATION.md + TACTICAL.md + PLAYBOOK.md per invoked theory, plus falsifier registry block, plus filtered INTERACTION_MATRIX with Shared Upstream Cause Warnings.
- Pass 4 reads severity discounts from falsifier registry entries, not from raw file text.
- No theory content from Inactive theories appears in any prompt.

These are structural tests — they verify inclusion and exclusion, not content quality.

### Layer 4: Run-Level Human Comparison — Judgment-Sensitive

Run one full pipeline cycle with old-format loader (via adapter), then one with new-format loader, on the **same briefing packet and the same run timestamp**.

- Activation scores: must match exactly (covered by Layer 1, but verified end-to-end here).
- Generation output: compared by structured human review. Are the hypotheses substantively similar? Same theories invoked? Same directional predictions? Same mechanism chains? Differences are expected (LLM non-determinism) but should be in phrasing, not substance.
- Elimination output: compared by structured human review. Same hypotheses killed? Same falsifiers cited? Same SURVIVED/WOUNDED/KILLED distribution?
- Conviction scores: compared numerically. Should be close (same severity discounts applied) but may differ slightly if generation/elimination produced different intermediate results.

**Minimum threshold:** 2-3 runs with human comparison before cutover. All three must show substantive equivalence.

## REGISTRY_INDEX.md

Generated to `theories/REGISTRY_INDEX.md` — a mechanical summary of the theory registry, produced as a loader byproduct after all 8 theory packages load successfully:

```
| theory_id | stability_class | phases | phase_names | falsifier_count (hard) | falsifier_count (soft) | indicator_count | current_status |
```

This is a convenience file for LLM orientation — the LLM reads REGISTRY_INDEX.md first to understand what theories exist before touching individual packages. It is regenerated on every loader run and should not be manually edited.

Scope: v8 produces this as a byproduct. It is not a blocking deliverable.

## Interface Contract Update

`ECONOMIC_THEORIES_INTERFACE_CONTRACT.md` must be updated to reflect the new file structure. The current contract defines a single `Theory Module` schema with fields like `activation_conditions`, `core_mechanism`, `predictions_when_active`, `downstream_implications`, `falsifiers`, and `metadata` — all assumed to live in one file.

The updated contract must specify:

1. The four-file package structure as the canonical format.
2. Required fields per file (as defined in `THEORY_REORGANISATION_SPEC.md`).
3. The FalsifierEntry schema (pre-joined object from Component 2).
4. The IndicatorOwnership schema (from Component 5).
5. The context_flags contract (surfaced to generator, excluded from scoring).
6. The INTERACTION_MATRIX format and filtering rules.

The updated contract replaces the current one. The old contract is archived alongside the old monolithic modules.

- The conviction scoring formula (Stage 1 → Stage 2 → Stage 3)
- The hypothesis lifecycle management (v7 — CONFIRM/UPDATE/RENEW/RETIRE/NEW taxonomy)
- The expression-level realization engine (v6 — payoff bands, freshness labels, realization caps)
- The sector falsifier appendices (v4)
- The regime annotation and channel tagging (v3)
- Any theory's actual economic content
- The generator's contract (7-9 hypotheses, mechanism + prediction + failure conditions)
- The evaluator's contract (SURVIVED / WOUNDED / KILLED + falsifier audit)

## What Is Explicitly Deferred

1. **Prompt budget optimisation.** Section-level filtering within files, content reduction for Adjacent theories, per-pass TACTICAL subsection routing. All deferred. Each pass gets the full files it needs.
2. **Data agent fetch orchestration by ownership type.** The data ownership classification is surfaced as metadata in v8. Using it to route the data agent's fetch strategy (mechanical → API, web-search → search tool, qualitative → skip) is a future enhancement.
3. **INTERACTION_MATRIX in Pass 4.** No interaction-based conviction adjustment. The matrix informs generation and elimination only.
4. **New theory addition workflow.** The reorganisation spec defines the format for new theories. Implementing a "new theory wizard" or validation tool is deferred.
5. **UI/Observatory changes.** The Theory Observatory currently displays information from monolithic modules. After v8, it should display from the new structure. Note what would need to change but do not implement in v8.
6. **Hypothesis thread index + run log.** These are v7 navigability layers for the hypothesis corpus. v8's REGISTRY_INDEX.md is the parallel navigability layer for the theory registry.

7. **Does not change the hypothesis lifecycle.** v7's CONFIRM/UPDATE/RENEW/RETIRE/NEW taxonomy is unaffected.
8. **Does not change the realization engine.** v6's payoff bands, freshness labels, and realization caps are unaffected.
9. **Does not change the frontend.** UI changes to display from packaged files are noted as deferred.
10. **Does not implement a new-theory-addition workflow.** The format is defined by the reorganisation spec; tooling is deferred.
